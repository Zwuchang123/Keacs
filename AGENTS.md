## 项目概览

- 项目名称：Keacs。
- 产品方向：完全离线、无需账号的本地个人记账 App。
- 产品要求：简单、轻量、便捷、易用。
- 产品目标：一个不认识字的人也能正常使用本软件进行记账。

## 执行规范

### 工作流程

- 多步骤任务使用短计划：

```text
 [步骤] -> 验证：[检查方式]
 [步骤] -> 验证：[检查方式]
 [步骤] -> 验证：[检查方式]
```

- 把请求改写成可验证目标：

```text
 “加校验” -> “先覆盖非法输入，再让测试通过”。
 “修 bug” -> “先复现失败，再修到通过”。
 “重构 X” -> “改前改后测试都能通过”。
```

- 先思考，再规划，最后执行。
- 参考必要的项目文档，只获取必要信息。
- 写代码前先明确当前理解、关键假设和验收目标；信息不足或存在分歧时先确认，不要擅自判断。
- 明显很小的任务可以简化流程，但不能跳过关键假设和结果验证。

### 常用命令

- PowerShell 读取中文前先执行：`chcp 65001 > $null; [Console]::OutputEncoding = [System.Text.Encoding]::UTF8`
- 完整自测：`.\gradlew.bat :app:compileDebugKotlin :app:assembleDebug :app:lintDebug :app:testDebugUnitTest :app:connectedDebugAndroidTest`
- 查看设备：`adb devices`
- 查看模拟器：`& "$env:ANDROID_HOME\emulator\emulator.exe" -list-avds`
- 启动模拟器：`& "$env:ANDROID_HOME\emulator\emulator.exe" -avd Pixel6_API34 -no-snapshot -no-audio -no-boot-anim`
- 安装并启动：`.\gradlew.bat :app:installDebug`，然后 `adb shell am start -n com.keacs.app/.MainActivity`
- 截图保存：`adb shell screencap -p /sdcard/page.png`，然后 `adb pull /sdcard/page.png docs\screenshots\iteration1\page.png`
- 检查联网权限：`Select-String -Path "app\src\main\AndroidManifest.xml" -Pattern "INTERNET|ACCESS_NETWORK_STATE|uses-permission" -SimpleMatch`
- 提交前查看：`git status --short`

### 开发要求

- 必须严格按照 `docs\plan.md` 中规划的迭代路线执行，
- 只做当前需求，优先选择更简单直接的方案，禁止为未来假设提前设计或过度抽象。
- 单次使用的逻辑不要抽象成框架、配置层或通用工具；不为低概率场景堆复杂防御。
- 前端必须考虑UI设计的审美规范，禁止简化设计要求，简单交付功能！

### 前端开发规范

- 前端界面禁止加入开发说明、调试说明或解释性文本，仅保留必要的用户界面元素。
- 前端开发必须使用项目内的 `impeccable` 技能。
- 以`docs\prd.md`和 `docs\design.md`为准，参考 `docs\UI-refercence.png`和目前已开发好的组件中已有的风格，完成前端开发；
- 禁止遗漏本轮迭代涉及的页面、功能组件、输入与选择组件、图标、状态反馈、提示提醒、弹窗、空态、错误态等任何内容；
- 开发完成后必须启动模拟器对所有页面、功能组件、输入、选择等组件完成自检，参考下面UI自检清单
- 截图完成后需主动通知用户确认；按修改意见进行调整确认‘
- 必须直到用户全部确认通过后，本轮迭代才算完成，完成后再主动创建git提交；最后更新`docs\plan.md`。

### UI自检清单

必须保证不存在以下问题才能进入下一步，自检不通过的必须主动修复通过：

- 是否存在文案错别字、缺字、截断、语义不清和与需求不一致；
- 是否存在字体大小、字重、颜色、圆角、阴影、边框、图标风格不统一；
- 是否存在间距、对齐、层级、留白、点击区域和布局比例明显不协调；
- 是否存在按钮不可点、点击无反馈、重复点击异常、返回异常和弹窗无法关闭；
- 是否存在输入框、选择器、日期时间、金额、开关等组件状态错误或默认值异常；
- 是否存在禁用态、加载态、空态、错误态、提示文案和成功反馈缺失；
- 是否存在列表截断、内容重叠、遮挡、溢出、滚动异常和底部内容被系统栏遮住；
- 是否存在颜色过杂、重点不清、视觉噪音过多、页面发灰发飘或廉价感明显的问题；
- 是否与 `docs\design.md`、设计图、已有组件样式和当前迭代截图保持一致；

### 代码规范

- 保持 UI 样式、组件、交互和体验一致，禁止擅自扩展或臆造能力。
- 代码必须简单、清晰、可维护，先保证正确性，再考虑性能，禁止过度设计。
- 命名准确清晰，职责单一，边界明确，结构统一，风格一致，并遵循现有代码风格。
- 控制复杂度，避免深层嵌套、重复逻辑和重复实现，优先复用现有能力。
- 只修改与当前任务直接相关的代码，不顺手重构、不修改无关格式、注释或行为；发现无关问题可以提示，但不要借机扩改。
- 只清理自己改动引入的无用代码；禁止硬编码，配置必须统一管理；禁止修改任务范围外代码。
- 保持向后兼容，覆盖边界条件、异常输入和失败场景，禁止静默失败或吞异常。
- 禁止暴露敏感信息，所有输入默认不可信。
- 代码必须包含必要的中文注释，但只说明“为什么”，避免冗余；
- 单文件不要超过300行，除非显著有利于开发实现（必须主动说明理由），任何情况下都严禁超过800行
- 每一处改动都必须能说明其与当前任务的直接关系。
- 跟随现有风格，即使你个人会用另一种写法；

### 测试问题修复

- 修复测试问题时，禁止改动无关代码，保证页面布局正常；逐条验证通过后再提交，并更新 `docs\test-issue-tracking.md`
- 用户在对话中提出的修复问题，需主动登记到 `docs\test-issue-tracking.md`

### 测试要求

每轮迭代完成后，必须按如下要求完成自测；未全部通过前不得进入下一步（非代码开发类任务适用本要求）：

- 必测项：Lint、类型检查、单元测试、集成测试、UI 测试、E2E 测试、性能测试、安全测试。
- 覆盖范围：主路径、边界值、异常场景、状态流转、兼容性。
- 主路径：覆盖从入口到结果的完整正常流程。
- 边界值：覆盖空值、最大值、最小值和非法输入。
- 异常场景：覆盖接口异常、权限问题、保存失败、超时、断网和重试恢复。
- 状态流转：覆盖返回、多步骤切换、重复进入、重复提交和幂等性。
- 兼容性：覆盖不同设备、系统版本、分辨率和大字体场景。
- 有失败项时，先修复，再重新完成完整自测。
- 不用每次都测断网启动和无网络权限检查，按需测试即可。

### 提交要求

- 对于开发或代码修复任务，任务确认完成后（需经用户确认才算完成），需主动创建 git commit，并使用中文提交说明，并且在完成后将提交记录补充到 `docs\auto-commit-log.md`。
- 更新`docs\auto-commit-log.md`和`docs\plan.md`的不用做主动提交

### 禁止事项

- 禁止未确认需求、假设或分歧就直接开始写代码。
- 禁止把小修复扩大成新架构、通用能力或大范围重构。
- 禁止修改相邻代码来“顺手优化”，导致出现无关 diff。
- 禁止没有完成验证就给出“应该可以”的结论。
- 禁止遇到不确定性时隐藏假设，必须明确说明。

### 完成标准

- 假设已明确说明，或已通过上下文和代码验证。
- 改动范围足够小，没有无关修改。
- 验收目标明确，且已完成对应检查和验证。
- 如果有检查无法执行，必须说明原因、影响和风险。
