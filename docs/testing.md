# 测试要求

## 1. 原则

- 先判断任务类型和影响面，再选择最低成本、能证明结果的检查。
- 优先小而快的测试；只有风险需要时才扩大到集成、UI、E2E 或性能。
- 只验证当前改动直接相关的行为，不为了“看起来完整”跑无关测试。
- Bug 修复必须先复现，再修复，再回归；无法复现时说明原因和风险。
- AI 生成或修改代码后，必须用命令或明确的手工检查验证，不用“应该可以”收尾。
- 有失败项时，先修复，再重新运行受影响范围内的检查。

## 2. 测试分层

| 层级         | 用途                              | 触发条件                  |
| ---------- | ------------------------------- | --------------------- |
| 文档检查       | 验证内容、链接、索引、规范一致性                | 只改文档或配置说明             |
| 静态检查       | 编译前发现格式、Lint、类型和明显错误            | 任何代码改动                |
| 小测试        | 验证单个函数、UseCase、规则或 ViewModel    | 逻辑、校验、计算、状态变更         |
| 集成测试       | 验证 Room、Repository、导入导出、网络边界等协作 | 存储、迁移、网络、跨层流程         |
| UI 自检      | 验证页面展示、交互、状态和适配                 | UI、布局、文案、导航、交互变更      |
| E2E / 设备测试 | 验证完整用户链路                        | 大功能、发布前、关键链路或高风险改动    |
| 性能 / 安全检查  | 验证耗时、资源、权限、敏感信息                 | 涉及性能、权限、密钥、日志、上传或安全边界 |

## 3. 按影响面补测

- 输入、校验、计算、状态判断：补测空值、最大值、最小值、非法输入和临界状态。
- 网络、权限、存储、保存、超时、重试：补测失败、拒绝、断网、重试和恢复。
- 数据库、迁移、导入导出：补测旧数据、空数据、重复数据、事务失败和兼容版本。
- 页面切换、返回、重复进入、重复提交、多步骤流程：补测状态流转和防重复执行。
- UI、布局、文案、字体、系统适配：补测可读性、遮挡、溢出、反馈状态和风格一致性。
- AI 助手写入：补测预览、确认、取消、重复确认、候选不唯一、本地校验失败和离线不受影响。
- 密钥、日志、上传、隐私：补测敏感信息不明文展示、不写日志、不越权上传。

## 4. Android 常用命令

```bash
./gradlew :app:compileDebugKotlin
./gradlew :app:assembleDebug
./gradlew :app:lintDebug
./gradlew :app:testDebugUnitTest
./gradlew :app:connectedDebugAndroidTest
```

完整自测：

```bash
./gradlew :app:compileDebugKotlin :app:assembleDebug :app:lintDebug :app:testDebugUnitTest :app:connectedDebugAndroidTest
```

本机无设备或模拟器时，可跳过 `connectedDebugAndroidTest`，但必须说明影响和风险。

## 5. 报告格式

- 只报告执行了哪些检查、通过/失败数量、无法执行项及风险。
- 不逐条列出用例，除非用户要求或有失败需要定位。
- 代码未改动时，明确说明未执行代码测试。
- 若选择不跑更大范围测试，说明依据，例如“本次只改文档”或“未涉及 UI/存储/网络”。
