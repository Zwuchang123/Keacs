## 1. 产品定位

- 产品名称：Keacs。
- 面向对象：个人自用记账用户。
- 核心目标：简单、轻量、离线可用，打开就能记。
- 产品原则：优先满足快速记账、看懂收支、稳定保存，不引入复杂财务模型。

当前实现以 Android 版本为准。基础记账能力不依赖账号、云同步、多人协作和复杂预算体系。Agent 助手是可选在线能力，首版不包含账号登录和云同步，用户不开启在线助手时，原有离线功能必须完整可用。

## 4. 使用流程

### 4.1 首次使用

用户首次打开 App 时，先看到欢迎页。欢迎页展示：

- 产品名称“轻记账”
- 副标题“轻量 · 简单 · 离线记账”
- 插画
- “开始记账”按钮
- “基础记账默认保存在本设备，启用在线助手后会使用云端智能能力”提示

点击“开始记账”后写入本机标记，随后进入首页。

### 4.2 再次打开

同一设备后续再次打开 App 时，不再展示欢迎页。应用完成本地初始化后，直接进入首页。

### 4.3 日常使用

1. 在首页查看选中月份的结余、收入、支出和每日账单明细。
2. 在底部点击“新增”进入记账页。
3. 选择“支出”“收入”或“转账”。
4. 选择分类或账户，输入金额、日期、备注后保存。
5. 在统计页查看趋势和排行，在“助手”页通过对话处理记账和查账，在“我的”页查看当月概览和资产负债。
6. 在“我的”页管理分类、账户或进行备份迁移。
7. 可从 Excel 添加已有账目，也可设置周、月、年的定时记账。

### 4.4 在线助手使用

1. 用户先在设置页启用在线助手，并选择模型服务模式。
2. 用户从底部导航进入“助手”页。
3. 用户输入自然语言，例如“昨天午饭 18 微信”或“这个月餐饮花了多少”。
4. 助手根据问题、历史对话、用户习惯和账本数据组织上下文。
5. 复杂问题进入状态化处理，页面展示“正在理解、正在读取账本、正在生成结果、等待确认”等进度。
6. 查询类问题直接展示回答，并说明本次统计或分析的数据范围。
7. 写入类问题先展示操作预览，必要时可让用户补充信息或选择候选账目。
8. 用户确认后，应用按现有记账规则保存、修改或删除数据。
9. 用户取消时，不写入任何数据，并结束本次待确认任务。
10. 用户离开页面后再次进入，未完成的待确认任务仍可继续处理。
11. 用户可对回答点赞、点踩或重新生成，系统用于改进后续回答。
12. 在线助手不可用时，页面展示原因和下一步，不影响本地记账。

## 5. 首页

首页用于按月查看概览和每日账单明细。

### 5.1 展示内容

- 选中月份
- 选中月份结余
- 选中月份收入
- 选中月份支出
- 按日期分组的每日账单明细

### 5.2 月份筛选

- 支持查看最近 10 年内的月份，且不能选择未来月份。
- 切换月份后，首页概览和每日账单明细同步刷新。

### 5.3 每日账单明细

- 列表按日期分组，组内按时间倒序展示。
- 每日分组只显示月日和周几。
- 每日分组展示当天支出合计。
- 支持点击单条记录进入账目详情。
- 没有记录时展示空态。

### 5.4 当前范围说明

首页不直接展示总资产、总负债和净资产。

## 6. 助手页

助手页是 Agent 助手的一级入口，用于通过对话完成记账、查账、改账、删账、定时记账、消费复盘和轻量建议。

助手页是独立开发任务的核心页面。开发时优先阅读本章节。

### 6.1 页面状态

助手页需要覆盖以下状态：

- 未启用：提示用户到设置页启用在线助手。
- 未配置：提示用户选择模型服务模式；自定义模式下提示填写访问地址和 API Key。
- 空会话：展示简短欢迎语和常用示例。
- 正常对话：展示用户消息、助手回复和操作卡片。
- 发送中：展示轻量加载状态，禁止重复发送同一条消息。
- 任务处理中：展示当前阶段，如正在理解、读取账本、生成方案或校验结果。
- 需要补充上下文：展示将读取的数据范围，并继续处理当前任务。
- 等待确认：展示待执行操作，等待用户确认或取消。
- 待确认恢复：用户重新进入页面时，继续展示未完成的确认卡片。
- 回答可反馈：助手回复旁提供轻量点赞、点踩和重新生成入口。
- 执行成功：展示结果，并刷新相关本地数据。
- 执行失败：说明失败原因，保留用户输入和上下文。
- 服务器不可用：提示网络或服务器问题，不影响本地记账。
- 长对话提醒：对话内容过长时提示用户清空，避免旧上下文影响判断。

### 6.2 动态引导

空会话和已有会话都需要展示轻量引导，但引导词不能固定为少量静态示例。

引导词生成规则：

- 根据当天日期生成，例如月初、月末、周末、节假日前后展示不同建议。
- 根据最近对话生成，例如用户刚问过餐饮，就可引导继续查看交通或本周趋势。
- 根据账本状态生成，例如本月有大额支出，可引导用户复盘。
- 根据用户习惯生成，例如经常晚上补记账，可引导“补记今天的支出”。
- 引导词保持 2 到 4 条，每条短句，不占用主要对话空间。

用户点击引导词后，将内容填入输入框或直接发送，具体交互按实现成本选择一种即可。

### 6.3 对话输入

首版只支持文字输入。

输入规则：

- 输入不能为空。
- 发送中禁止重复点击发送。
- 发送失败后保留输入内容。
- 单条输入需要限制最大长度，避免一次发送过长内容。
- 输入框需要支持多行自动换行，内容过多时不横向挤压。
- 不支持语音、图片、票据识别、文件上传。

用户点击发送前，不上传任何账本数据。

### 6.4 消息类型

助手页至少支持以下消息类型：

- 用户消息：展示用户输入的原文。
- 助手文本回复：用于回答查账、解释结果和追问。
- 操作预览卡片：用于展示待确认的新建、修改、删除、批量处理和定时记账。
- 候选选择卡片：用于展示多条可能匹配的账目，让用户选择。
- 任务进度消息：用于展示助手正在处理的阶段，不展示内部节点、工具名或提示词。
- 上下文范围提示：用于说明本次会读取哪些账本范围，例如本月账目、最近 7 天账目或账户列表。
- 反馈操作：助手回复旁展示点赞、点踩和重新生成小图标，不占用主要视觉空间。
- 执行结果消息：用于展示保存成功、修改成功、删除成功或执行失败。
- 错误消息：用于展示网络、配置、模型和本地校验异常。

消息按时间从上到下展示，进入页面后默认定位到最新消息。

消息展示规则：

- 消息卡片根据内容自适应宽高，只设置最大宽度。
- 助手回复支持换行和重点加粗，避免整段文字难以阅读。
- 发送中和回复完成后展示本次响应耗时。
- 点赞、点踩和重新生成只作为轻量图标出现，不能打断主流程。

### 6.5 记账理解规则

助手从用户输入中识别以下信息：

- 记账类型：支出、收入、转账。
- 金额。
- 日期。
- 分类。
- 账户。
- 备注。

默认规则：

- 未说日期时，默认使用今天。
- 日期不能是未来日期。
- 金额必须大于 0，最多保留 2 位小数。
- 能明确识别分类时自动带出分类。
- 收入和支出未说账户时，优先使用设置中的默认记账账户；没有默认账户时可留空。
- 转账必须识别转出账户和转入账户；缺失时必须追问。
- 无法明确识别金额、类型或分类时，必须追问，不能直接保存。
- 用户原话中有可读备注时，可写入备注；没有备注时留空。

示例：

- “昨天午饭 18 微信”识别为支出，分类为餐饮，账户为微信，日期为昨天。
- “今天工资 8000 到银行卡”识别为收入，分类为工资，账户为银行卡。
- “从微信转 200 到银行卡”识别为转账，转出账户为微信，转入账户为银行卡。

### 6.6 查账和分析

助手支持回答账本相关问题。

支持范围：

- 月度收入、支出、结余。
- 分类支出或收入汇总。
- 最近一段时间的大额支出。
- 指定账户余额。
- 消费习惯复盘。
- 预算提醒。
- 省钱建议。

查账优先按用户问题选择数据范围，不固定只看本月。常见范围包括今天、昨天、本周、最近 7 天、最近 30 天、最近 3 个月、最近半年、最近一年、今年、去年、本月、上月和全部账单。

启用官方在线助手后，后端允许保存和使用全部账本数据。单次回答仍按问题选择合适上下文：长期分析可使用全部账本，单日、单周、单月或单年问题优先使用对应范围和统计摘要，避免上下文过大影响回答质量。

助手回答时需要说明数据范围，例如“按 2026 年 5 月统计”或“按最近 7 天统计”。

### 6.7 改账和候选选择

用户要求修改账目时，助手需要先定位候选账目。

候选账目展示内容：

- 日期
- 记账类型
- 金额
- 分类
- 账户
- 备注

规则：

- 只匹配到 1 条时，可直接生成修改预览。
- 匹配到多条时，必须让用户选择。
- 匹配不到时，提示用户补充日期、金额、分类或账户。
- 修改后仍必须符合记账规则。

示例：

- “把昨天午饭改成 20”
- “上周那笔打车分类改成交通”
- “把工资账户改成银行卡”

### 6.8 删账

用户要求删除账目时，助手需要先定位候选账目并展示删除预览。

规则：

- 删除必须二次确认。
- 删除预览必须展示将删除的账目数量和明细。
- 匹配到多条时，必须让用户选择或确认批量删除。
- 用户取消后，不删除任何数据。

示例：

- “删掉昨天重复的午饭”
- “删除今天那笔 18 元支出”

### 6.9 批量处理

助手支持有限批量处理。

支持范围：

- 批量修改分类。
- 批量修改账户。
- 批量追加或修改备注。

规则：

- 批量处理必须展示影响条数和明细列表。
- 批量处理必须用户确认后执行。
- 影响范围无法确定时必须追问。
- 首版不支持批量新增大量账目。

示例：

- “把上周所有打车都归到交通”
- “把本月备注里有咖啡的支出改成餐饮”

### 6.10 定时记账

助手支持创建、修改、停用定时记账。

创建和修改定时记账时，需要展示：

- 收支类型
- 重复周期
- 金额
- 分类
- 账户
- 转出账户
- 转入账户
- 下次生成时间
- 备注
- 是否启用

定时记账仍沿用 `7.8 定时记账` 的规则。

示例：

- “每月 1 号房租 2500，从银行卡支出”
- “每周一记一笔早餐 10 元”
- “停用房租定时记账”

### 6.11 操作确认

所有会改变本地数据的操作都必须用户确认后执行。

写入类操作包括：

- 新增账目
- 修改账目
- 删除账目
- 批量修改
- 创建定时记账
- 修改定时记账
- 停用定时记账

操作预览必须展示：

- 操作类型
- 影响对象
- 金额
- 日期
- 分类
- 账户
- 备注
- 影响条数
- 风险提示

用户确认后，应用按现有业务规则执行。用户取消后，不写入任何数据。

确认交互规则：

- 操作预览卡片必须直接提供“确认执行/确认删除”和“取消”按钮。
- 操作预览需要有一次性操作标识，用户重复点击或页面重进后不能重复执行同一操作。
- 预览内容必须使用普通用户能看懂的账目格式，不能直接展示 JSON。
- 用户取消或确认后，需要移除对应待确认动作，避免重复执行。
- 如果用户发现预览有误，可以取消后继续补充说明；首版不要求在确认卡片内直接编辑字段。

### 6.12 轻量建议边界

助手可以提供：

- 消费习惯复盘。
- 预算提醒。
- 支出异常提醒。
- 省钱建议。
- 分类使用建议。

助手禁止提供：

- 投资建议。
- 借贷建议。
- 保险建议。
- 税务、法律、医疗等专业建议。
- 承诺收益或预测收益。

遇到禁止范围时，助手需要说明无法提供该类建议，并可引导用户回到账本记录和消费复盘。

### 6.13 异常处理

助手需要处理以下异常：

- 未启用在线助手。
- 自定义模式未配置访问地址。
- 自定义模式未配置 API Key。
- 服务器不可达。
- API Key 无效或模型服务拒绝访问。
- 模型调用失败。
- 模型返回内容无法解析。
- 模型响应较慢。
- 本地业务校验失败。
- 候选账目不唯一。
- 分类或账户不存在。
- 日期为未来时间。
- 金额不合法。

异常提示必须说明原因和下一步，不使用错误码作为主要文案。

模型响应较慢或返回不稳定时，不应只展示“模型服务不可用”。能用本地规则判断的常见记账和查账请求，需要先给出可读结果或待确认预览；不能可靠判断时，再提示用户补充信息或稍后重试。

### 6.14 会话保留与清理

- 助手需要保留最近对话，用户退出后再次进入仍能看到原对话。
- 启用在线助手后，官方服务允许保存用户对话、账本数据、反馈结果和偏好习惯，用于个性化体验和自动优化。
- 请求模型时需要按任务选择上下文，不能把长期历史无控制地塞进同一次模型调用。
- 用户可以手动清空对话。
- 对话过长时，系统应自动摘要、归档或分段检索，避免旧内容影响新的判断。

### 6.15 状态化助手能力

助手不按单次问答理解，而按一次“任务”处理。一次任务可以包含理解、读取本地上下文、生成回答、展示预览、等待确认、执行反馈和完成。

前端需要支持：

- 展示任务阶段，让用户知道助手正在做什么。
- 展示本次使用的数据范围，例如“读取本月账目和分类账户”。
- 对长耗时任务展示流式进度，避免页面长时间无反馈。
- 对需要确认的写入任务保留待确认状态，用户离开后再回来仍可处理。
- 对多候选账目展示选择卡片，用户选择后继续当前任务。
- 对执行失败的任务保留可读失败原因，并允许用户基于原问题继续补充。

前端不展示：

- 后端内部节点名。
- 模型提示词。
- 工具调用原始参数。
- 账本数据的原始 JSON。

任务状态必须以用户能理解的文案表达，例如“正在整理本月账目”“需要你确认这笔修改”“保存失败，请检查分类是否还可用”。

### 6.16 反馈与自我优化

助手需要支持轻量反馈，让系统根据真实使用情况持续变好。

前端反馈能力：

- 每轮助手回答旁展示点赞、点踩和重新生成小图标。
- 点赞表示回答有帮助。
- 点踩表示回答不满意，首版可不要求用户填写原因。
- 重新生成表示用户希望换一种回答或重新规划。
- 反馈图标要弱展示，不抢占对话正文和确认卡片的视觉重点。

系统优化能力：

- 根据用户反馈、重新生成、取消确认、执行失败和追问情况，判断回答是否有效。
- 根据长期对话和账本数据学习用户习惯，例如常用账户、常用分类、表达方式、复盘偏好和提醒偏好。
- 对同一用户优先使用其个人习惯；对整体产品可汇总匿名趋势优化提示词、工具路由和上下文选择策略。
- 当用户经常改正分类或账户时，后续自动提高这些偏好的优先级。
- 当某类问题经常点踩或重新生成时，系统应降低对应回答模板或路由策略的使用优先级。

边界：

- 自我优化不能绕过写入确认。
- 自我优化不能自动修改用户账本。
- 个性化偏好只能用于提升助手理解和回答质量，不改变基础记账规则。

### 6.17 动态引导词

助手引导词需要动态生成，不使用固定的一组文案长期展示。

动态来源：

- 日期：今天、昨天、本周、月初、月末、年末、节假日等。
- 账本：近期支出、收入、异常大额、常用分类、常用账户、定时记账状态。
- 对话：最近问题、未完成确认、用户刚刚点踩或重新生成的内容。
- 用户偏好：喜欢简短回答、喜欢消费复盘、常用自然语言表达。

展示要求：

- 每次展示 2 到 4 条。
- 每条文案短、具体、可直接点击。
- 优先生成与当前用户最可能相关的问题。
- 不在确认卡片和错误提示上方强插引导词。
- 引导词可以在会话底部或输入框上方轻量展示。

## 7. 记账需求

### 7.1 记录类型

当前版本支持三类记录：

- 支出
- 收入
- 转账

### 7.2 通用字段

三类记录的共同输入包括：

- 金额
- 日期
- 备注

规则：

- 金额必须大于 0 才能保存。
- 日期只能选择当前或过去时间，不能选择未来日期。
- 备注为选填。
- 编辑记录时沿用原始数据进行回填。

### 7.3 支出记录

字段：

- 金额
- 支出分类
- 支出账户
- 日期
- 备注

规则：

- 分类必填。
- 账户可不选；如已在设置中配置默认记账账户，进入时会优先带出。
- 保存后对应账户余额减少。
- 删除或编辑后，余额和统计自动修正。

### 7.4 收入记录

字段：

- 金额
- 收入分类
- 收入账户
- 日期
- 备注

规则：

- 分类必填。
- 账户可不选；如已在设置中配置默认记账账户，进入时会优先带出。
- 保存后对应账户余额增加。
- 删除或编辑后，余额和统计自动修正。

### 7.5 转账记录

字段：

- 金额
- 转出账户
- 转入账户
- 日期
- 备注

规则：

- 转出账户必填。
- 转入账户必填。
- 转出账户和转入账户不能相同。
- 转账不计入收入或支出统计，只影响账户余额。
- 删除或编辑后，余额自动修正。

### 7.6 金额输入

当前金额键盘支持：

- 数字输入
- 小数点
- 退格
- `+` 和 `-` 连续计算

规则：

- 最多保留 2 位小数。
- 输入总长度上限为 11 个字符。
- 表达式结果必须大于 0 才可保存。
- 例如 `10+2.50` 可以直接计算为 `12.50`。

### 7.7 账目详情

账目详情页当前展示：

- 记录类型/分类信息
- 金额
- 账户信息
- 账目日期
- 备注

并提供：

- 编辑记录
- 删除记录

当前详情页不展示创建时间和更新时间。

### 7.8 定时记账

定时记账用于重复发生的收入、支出和转账。

字段：

- 收支类型
- 重复周期（周、月、年）
- 金额
- 分类
- 账户
- 转出账户
- 转入账户
- 生成时间
- 备注
- 是否启用

规则：

- 定时记账支持收入、支出和转账。
- 到达生成时间后，应用本地生成对应账目，并推进到下一次时间。
- 生成时间固定为 09:00，不提供自定义小时。
- 按周、按月、按年均支持多选配置。
- 收入和支出的账户可不选，分类必填。
- 转账必须选择转出账户和转入账户，且两个账户不能相同。
- 分类或账户不可用导致无法生成时，该定时记账会停用，避免反复失败。

## 12. 数据保存与迁移

### 12.1 保存原则

- 基础账本数据只保存在本地设备。
- 基础记账能力不依赖网络和账号系统。
- 未启用在线助手时，不上传任何账本数据。
- 启用官方在线助手后，允许上传并保存用户全部账本、对话、反馈和偏好数据，用于回答问题、个性化体验和自动优化。
- 模型单次调用仍需要做上下文管理，按任务选择、摘要或检索相关数据，避免无控制地塞入全部历史。
- 当前只支持人民币记账。

### 12.2 导出

当前支持导出单个 JSON 备份文件，备份内容包括：

- 分类
- 账户
- 记录
- 备份版本
- 导出时间
- App 版本号
- 余额符号规则

### 12.3 导入

当前只支持合并导入。

导入规则：

- 记录直接追加，不做去重。
- 分类按“方向 + 名称”复用已有项，不重复插入。
- 账户始终新增；如名称冲突，会自动重命名为“原名称（导入2）”等形式。
- 支持读取备份版本 1 和版本 2。
- 导入旧备份时，会把负债账户余额转换为当前负数规则。

### 12.4 Excel 添加账目

Excel 文件字段包括：

- 日期
- 收支类型
- 分类
- 账户
- 金额
- 备注

规则：

- 当前支持 `.xlsx` 文件。
- 收支类型支持收入和支出，不支持转账。
- 分类按当前已启用分类名称匹配；匹配不到时归到“其他”。
- 匹配不到分类时，备注追加“原分类：原名称”。
- 账户按当前已启用账户名称匹配；匹配不到时不选账户，避免整行失败。
- 空行会跳过；日期、类型、金额错误的行会跳过，其余可正常添加。

### 12.5 换机迁移

用户可在旧设备导出，在新设备导入，恢复分类、账户和记录数据。

### 12.6 在线助手数据边界

启用官方在线助手后，允许上传和保存：

- 用户全部账本数据。
- 用户完整对话记录。
- 用户点赞、点踩、重新生成、确认、取消和执行失败等反馈。
- 用户偏好习惯，例如常用账户、常用分类、表达习惯、复盘偏好和提醒偏好。
- 任务运行状态、模型输入输出、上下文摘要和优化分析结果。

仍然不能上传：

- 用户自定义 API Key；该 Key 只在调用用户配置的模型服务时使用，不上传到 Keacs 后端。
- 设备原始标识；上传前必须转换为不可逆摘要。
- 用户未启用在线助手时的本地账本数据。

后端允许保存的数据用于：

- 回答用户问题。
- 恢复长任务和历史对话。
- 建立用户个性化记忆。
- 优化提示词、上下文选择、工具路由和错误恢复。
- 统计模型质量、完成率、重试率和用户满意度。

上下文和资源管理规则：

- 后端可以保存全部数据，但单次模型调用必须按任务选择上下文。
- 长对话要自动摘要，旧内容可归档后按需检索。
- 大量账本分析优先使用统计摘要，必要时再读取明细。
- 对话、账本、偏好和反馈需要分层存储，避免每次请求都读取全部数据。
- 后端需要控制单次请求大小、模型输出长度、每日成本和长期存储体积。
- 用户手动清空对话时，前端会清空本机展示；官方服务端是否删除历史数据按后续账号与数据管理能力设计。

## 13. 核心业务规则

### 13.1 余额规则

- 资产账户以正数表示。
- 负债账户以负数表示。
- 收入记入账户时加余额。
- 支出记入账户时减余额。
- 转账按“转出减、转入加”处理。
- 账户性质不会改变收支方向，只影响余额符号展示。

### 13.2 历史一致性

- 新增、编辑、删除记录后，账户余额必须同步修正。
- 停用的账户和分类，其历史记录仍要可读。
- 统计结果始终以当前有效记录为准。

### 13.3 Agent 助手写入规则

- Agent 助手不能绕过现有记账规则。
- Agent 助手不能直接写入数据库。
- 所有新增、修改、删除、批量处理和定时记账操作，必须先展示预览。
- 用户确认后，应用按现有业务规则执行写入。
- 用户取消后，不写入任何数据。
- 写入成功后，账户余额和统计必须同步更新。
- 本地业务校验失败时，不执行写入，并提示失败原因。
- 每个待执行操作必须有一次性标识，已确认、已取消或已失败的操作不能再次执行。
- 后端或模型只能生成待确认操作，不能把写入结果伪装成已完成。
